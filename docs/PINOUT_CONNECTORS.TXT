1. CONN_MAIN (Power Input and CAN Bus)
   Type: B4B-XH-2 (2.54mm pitch)
-----------------------------------------------------------------------------------------
PIN | SIGNAL    | FUNCTION
-----------------------------------------------------------------------------------------
1   | +24V      | Main power supply (goes to the regulator and drivers)
2   | GND       | Common power ground
3   | CAN_H     | CAN Bus High (to TJA1057)
4   | CAN_L     | CAN Bus Low (to TJA1057)
    |
    | NOTE: whether this board carries a fixed 120-ohm termination
    |       resistor across CAN_H/CAN_L, or leaves it unpopulated/optional
    |       via a solder-bridge, isn't specified anywhere in this project's
    |       documentation yet. Doesn't matter if this board is always the
    |       last node on the bus - matters a lot if it's ever used as a
    |       middle node in a daisy chain, where a second fixed 120-ohm
    |       resistor would drag the bus impedance well below the 120-ohm
    |       (two terminators in parallel) the standard expects. Worth
    |       settling and documenting explicitly once the actual PCB
    |       decision is made.

3. CONN_OLED (Local Display Interface, Hardware I2C1)
   Type: JST4-B4B-PH-K (2.00mm pitch)
-----------------------------------------------------------------------------------------
PIN | SIGNAL    | FIRMWARE LOGIC FUNCTION
-----------------------------------------------------------------------------------------
1   | GND       | Ground
2   | +3.3V     | Display logic power

4   | SDA       | PA10 (Hardware I2C1_SDA, AF4)
    |           | R33/R34, 4.7kOhm pull-ups to +3.3V.
    |           | NOTE: this same I2C1 bus (PA9/PA10) also reaches the on-board

4. CONN_DRILL (Multi-function: Drill / Laser / 4-Wire Fan)
   Type: B6B-XH-2 (2.54mm pitch)
------------------------------------------------------------------------------------------------
PIN | SIGNAL       | DRILL MODE           | 3D MODE (DELTA FAN)              | LASER MODE
------------------------------------------------------------------------------------------------
1   | ENABLE/BRAKE | PB9 (BL4260 brake,   | PB6 (Extruder ENN - NOT the fan) | PB6 (Safety interlock)
    |              |  own dedicated pin,  |                                   |
    |              |  see note below)     |                                   |
2   | FGIN         | PA3 (Tachometer)     | PA3 (Tachometer / BLUE wire)     | NC
3   | FRIN         | PA4 (Rotation IN/OUT)| NC                               | NC
4   | PWM          | PA8 (Speed control)  | PA8 (25kHz real PWM / YELLOW wire,| PA8 (Emission power)
    |              |                      |  TIM1_CH1, same channel as       |
    |              |                      |  drill/laser, reconfigured per-  |
    |              |                      |  tool)                           |
5   | GND          | Ground               | Ground (BLACK wire)              | Ground
6   | +24V         | Power supply         | Power supply (RED wire)          | Power supply
    |              |                      |                                  |
    | NOTE: cross-checked against the Delta EFB0424VHD-CP0 datasheet:
    |   BLUE=FG (tachometer), YELLOW=PWM.
    | NOTE: the fan's "emergency stop" is a 1000ms CAN communication watchdog
    |       that zeroes the PWM autonomously - it does NOT touch PB6, since that
    |       line is already the extruder driver's enable in this same tool mode.
    | NOTE (NEEDS YOUR ATTENTION AT LAYOUT): Drill mode's ENABLE/BRAKE uses
    |       its own dedicated pin, PB9 - not shared with the stepper tools'
    |       TMC_ENN. This table still shows PB6 for
    |       3D/Laser mode on this same connector pin, matching the existing
    |       multi-mode pattern used throughout this document - but check
    |       whether that's actually what you want wired here: if CONN_DRILL's
    |       physical pin 1 ties BOTH PB6 and PB9 to the same net, make sure
    |       firmware never drives both as outputs at once for a mode where
    |       only one should be active (rather than assuming that's already
    |       handled safely). The simpler alternative is rewiring this pin to
    |       PB9 only, and moving the 3D-printer extruder-enable connection
    |       elsewhere (e.g. CONN_MOT, which already carries the real TMC2209
    |       driver lines for that tool) - whichever you decide, this is a
    |       trace-level call for your own Eagle work, not something assumed
    |       resolved by this document.

PIN | SIGNAL | FUNCTION
------------------------------------------------------------------------------------------------

2   | FG     | PA6 (Tachometer, EXTI6) - series-protected by R42
3   | PWM    | PA5 (TIM2_CH1/AF1, 25kHz) - series-protected by R41
4   | GND    | Ground
    |
    | NOTE: separate fan from the one on CONN_DRILL. This one cools the

8. CONN_LED2 (Lighting - Camera/Inspection Coaxial Ring)

-----------------------------------------------------------------------------------------
PIN | SIGNAL    | FIRMWARE LOGIC FUNCTION
-----------------------------------------------------------------------------------------
1   | GND       | Ground

9. CONN_EXPANSION (Add-on board header)
   Type: 20-pin, 1.27mm pitch, Würth eiCan series - see BOM.TXT
------------------------------------------------------------------------------------------------
PIN | SIGNAL         | FUNCTION
------------------------------------------------------------------------------------------------
1   | +24V           | Main rail passthrough
2   | +24V           | Main rail passthrough (second pin)
3   | GND            | Ground
4   | +5V            | Logic power
5   | EXP_TMC_STEP   | PB12 - STEP pulses. Universal to any STEP/DIR driver on the
    |                | expansion board (TMC2209 and TMC5160 both use this the same way)
6   | EXP_TMC_EN     | PB14 - driver enable, universal same as EXP_TMC_STEP
7   | EXP_I2C2_SCL   | PB10 - own bus, separate from the OLED's I2C1. R48 (4.7k to
    |                | +3.3V) populated as an external pull-up
8   | EXP_SPI_CS     | PA2 - bit-banged (software) SPI, not a hardware peripheral -
    |                | see the SPI note below for the reasoning.
9   | EXP_SPI_MISO   | PB8 - bit-banged, same bus as EXP_SPI_CS
10  | EXP_GPIO1      | PC14 - general purpose, EXTI-capable if a future add-on needs
    |                | a fast interrupt-driven input, e.g. an endstop
11  | EXP_TMC_DIAG0  | PC15 - a TMC5160's stall/fault diagnostic line, using the same
    |                | EXTI-capable headroom as EXP_GPIO1 - see the DIAG0 note below.
12  | EXP_SPI_MOSI   | PB15 - bit-banged, same bus as EXP_SPI_CS
13  | EXP_SPI_SCK    | PB2 - bit-banged, same bus as EXP_SPI_CS
14  | EXP_I2C2_SDA   | PB11. R44 (4.7k to +3.3V) populated as an external pull-up
15  | EXP_TMC_DIR    | PB13 - direction, universal same as EXP_TMC_STEP
16  | GND            | Ground
17  | +3V3           | Logic power
18  | GND            | Ground
19  | +24V           | Main rail passthrough (third pin, for higher-current add-ons)
20  | +24V           | Main rail passthrough (fourth pin, for higher-current add-ons)
    |                |
    | NOTE: 4 pins of +24V (was 2) and no no-connects remain, since a
    |       schematic revision reorganized this connector for exactly
    |       that: more current headroom on the main rail for a future
    |       add-on driving a real motor, and every pin actually doing
    |       something. Every MCU-side pin assignment is unchanged from
    |       before this reorganization - only which physical connector
    |       pin number each signal lands on moved.
    | NOTE - why bit-banged SPI, not a hardware peripheral: this chip's
    |       hardware SPI2 defaults to PB12/13/14/15, which this board already
    |       uses for EXP_TMC_STEP/DIR/EN/EXP_SPI_MOSI, so a hardware SPI2
    |       would mean re-routing those pins elsewhere first. SPI3's standard
    |       pins (PC10/PC11/PC12) aren't present on this chip's LQFP48
    |       package at all - confirmed against the official STM32F303xB/xC
    |       datasheet's per-package pin table, which shows the whole of
    |       Port D and several Port C pins simply absent on this footprint,
    |       only available on the larger LQFP64/LQFP100 variants of the same
    |       silicon. Bit-banging on 4 otherwise-unused pins sidesteps both
    |       constraints entirely, and this firmware already has a working
    |       precedent for bit-banged serial timing (the WS2812B ring driver
    |       on PB1) - SPI's per-bit timing requirements are far more
    |       forgiving than WS2812B's, so this is a comfortably proven
    |       approach for this codebase. TMC5160 configuration doesn't need
    |       anywhere near a hardware peripheral's top speed to work
    |       correctly. Confirmed against the TMC5160's own datasheet: it
    |       specifies SPI Mode 3 (CPOL=1, CPHA=1) explicitly by name, which
    |       the firmware's bit-banged implementation matches exactly - see
    |       ExpansionSPI_TransferByte() in STM32F303CC.C.
    | NOTE - EXP_TMC_DIAG0: EXP_GPIO1 and EXP_TMC_DIAG0 (pins 10/11) share
    |       the same design intent - general-purpose, EXTI-capable headroom
    |       for whatever fast interrupt-driven input a future add-on needs.
    |       DIAG0 is exactly that kind of signal, so it uses this headroom
    |       directly rather than needing dedicated pin budget of its own.
    |       EXP_GPIO1 remains available as genuinely general-purpose,
    |       unallocated headroom for anything else. DIAG0 is currently
    |       exposed as a simple polled CAN query (`0x182`/`0x183` - see
    |       CANBUS.TXT) rather than a hardware EXTI interrupt the way the
    |       scan probe's own PB3 is - a natural option once a concrete
    |       expansion board makes clear what latency actually matters here.
    | NOTE: an I2C ADC/DAC chip or a port-expander chip on the add-on board
    |       both hang off EXP_I2C2_SCL/EXP_I2C2_SDA like any other I2C device
    |       - no dedicated pins needed for those specifically. See
    |       PINOUT.TXT Block 10 and the README's "Expansion Connector"
    |       section.
    | NOTE - TMC2209 and TMC5160 sharing: the expansion board is expected to
    |       carry one or the other, not necessarily both - a TMC2209 uses
    |       only EXP_TMC_STEP/DIR/EN plus its own single-wire UART
    |       (TMCUART, half-duplex), never the 4 EXP_SPI_* pins, which have
    |       no meaning to that chip at all. Since the two populate options
    |       are mutually exclusive on any given board, EXP_SPI_MOSI (pin 12)
    |       doubles as a natural home for a TMC2209's single UART line when
    |       that's the chip actually populated - no separate dedicated UART
    |       pin is needed, since one is never required at the same time as
    |       the SPI bus is. A board wanting both chips populated together
    |       can still wire EXP_TMC_STEP/DIR/EN to both drivers in parallel
    |       like any shared STEP/DIR bus, but the TMC2209 would then need
    |       its own separate UART pin rather than sharing pin 12 with the
    |       TMC5160's MOSI.

========================================================================================
FREE MCU PINS (not on any connector): none - every GPIO pin on this design
is committed to something. PB1 drives CONN_LED2. PA8 is the drill/laser
PWM output (see the table above). PC13 carries ID4, PB9 carries the
drill's dedicated brake, and PB10-PB14/PC14-PC15 feed CONN_EXPANSION
above, as does the EXP_SPI_* bus (PA2/PB2/PB8/PB15). Port D is not
present at all on this chip's LQFP48 package (confirmed against the
official STM32F303xB/xC datasheet's per-package pin table) - not a
source of any free pins on this board, regardless of what's used
elsewhere.
